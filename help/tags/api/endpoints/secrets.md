# Secrets endpoint

A secret is a resource that exists only within event forwarding properties (properties with a `platform` attribute set to `edge`). They allow event forwarding to authenticate to another system for secure data exchange.

This guide shows you how to make calls to the `/secrets` endpoint in the Reactor API. For a detailed explanation of the different secret types and how to use them, please refer to the high-level overview on [secrets](../guides/secrets.md) before returning to this guide.

## Getting started

The endpoint used in this guide is part of the [Reactor API](https://www.adobe.io/apis/experienceplatform/home/api-reference.html#!acpdr/swagger-specs/reactor.yaml). Before continuing, please review the [getting started guide](../getting-started.md) for important information regarding how to authenticate to the API.

## Retrieve a list of secrets for a property {#list-property}

You can list the secrets that belong to a property by making a GET request.

**API format**

```http
GET /properties/{PROPERTY_ID}/secrets
```

| Parameter | Description |
| --- | --- |
| `{PROPERTY_ID}` | The ID of the property whose secrets you want to list. |

{style="table-layout:auto"}

```shell
curl -X GET \
```

## Create a secret {#create}

You can create a secret by making a POST request.

>[!NOTE]
>
>When you create a new secret, the API returns an immediate response that contains information for that resource. At the same time, a secret exchange task is triggered to test that the credential exchange is functional. This task is processed asynchronously and updates the secret's status attribute to `succeeded` or `failed` depending on the outcome.

**API format**

```http
POST /properties/{PROPERTY_ID}/secrets
```

| Parameter | Description |
| --- | --- |
| `{PROPERTY_ID}` | The ID of the property that you want to define the secret under. |

{style="table-layout:auto"}

**Request**

```shell
curl -X POST \
  https://reactor.adobe.io/properties/PR9eff664dc6014217b76939bb78b83976/secrets \
  -H 'Authorization: Bearer {ACCESS_TOKEN}' \
  -H 'x-api-key: {API_KEY}' \
  -H 'x-gw-ims-org-id: {ORG_ID}' \
  -H 'Accept: application/vnd.api+json;revision=1' \
  -H 'Content-Type: application/vnd.api+json' \
  -d '{
        "data": {
          "attributes": {
            "name": "Example Secret",
            "type_of": "simple-http",
            "credentials": {
              "username": "example_username",
              "password": "pass12345"
            }
          },
          "relationships": {
            "environment": {
              "data": {
                "id": "EN04cdddbdb6574170bcac9f470f3b8087",
                "type": "environments"
              }
            }
          },
          "type": "secrets"
        }
      }'
```

| Property | Description |
| --- | --- |
| `name` | A unique, descriptive name for the secret. |
| `type_of` | The type of authentication credential the secret represents. Has three accepted values:<ul><li>`token`: A token string.</li><li>`simple-http`: A username and password.</li><li>`oauth2`: Credentials conforming to the OAuth standard.</li></ul> |
| `credentials` | An object that contains the credential values for the secret. Depending on the `type_of` attribute, different properties must provided. See the section on [credentials](../guides/secrets.md#credentials) in the secrets guide for details on the requirements for each type. |
| `relationships.environment` | Each secret must be associated with an environment when it is first created. The `data` object within this property must contain the `id` of the environment the secret being assigned to, along with a `type` value of `environments`. |
| `type` | The type of resource being created. For this call, the value must be `secrets`. |

{style="table-layout:auto"}

**Response**

A successful response returns the details of the secret. Note that depending on the type of secret, some properties under `credentials` may be hidden.

## Test an `oauth2` secret {#test}

>[!NOTE]
>
>This operation can only be performed on secrets with a `type_of` value of `oauth2`.

You can test an `oauth2` secret by including its ID in the path of a PATCH request. The test operation performs an exchange and includes the authorization service response in the `test_exchange` attribute in the secret's `meta` object. This operation does not update the secret itself.

**API format**

```http
PATCH /secrets/{SECRET_ID}
```

| Parameter | Description |
| --- | --- |
| `{SECRET_ID}` | The ID of the `oauth2` secret that you want to test. |

{style="table-layout:auto"}

**Request**

```shell
curl -X PATCH \
  https://reactor.adobe.io/secrets/SE6c15a7a64f9041b5985558ed3e19a449 \
  -H 'Authorization: Bearer {ACCESS_TOKEN}' \
  -H 'x-api-key: {API_KEY}' \
  -H 'x-gw-ims-org-id: {ORG_ID}' \
  -H 'Accept: application/vnd.api+json;revision=1' \
  -H 'Content-Type: application/vnd.api+json' \
  -d '{
        "data": {
          "attributes": {
            "type_of": "oauth2"
          },
          "meta": {
            "action": "test"
          },
          "id": "SE6c15a7a64f9041b5985558ed3e19a449",
          "type": "secrets"
        }
      }'
```

| Property | Description |
| --- | --- |
| `attributes` | Must contain a `type_of` property with a value of `oauth2`. |
| `meta` | Must contain an `action` property with a value of `test`. |
| `id` | The ID of the secret you are testing. This must match the ID provided in the request path. |
| `type` | The type of resource being operated on. Must be set to `secrets`. |

{style="table-layout:auto"}

## Retry a secret {#retry}

Retrying a secret is the action of manually triggering the secret exchange. You can retry a secret by including its ID in the path of a PATCH request.


| --- | --- |

```shell
curl -X PATCH \
```

## Reauthorize an `oauth2-google` secret {#reauthorize}

Each `oauth2-google` secret contains a `meta.authorization_url_expires_at` property that indicates when the authorization URL will expire. After this time, the secret must be reauthorized in order for it to renew the authentication process.

To reauthorize an `oauth2-google` secret, make a PATCH request for the secret in question.

**API format**

```http
PATCH /secrets/{SECRET_ID}
```

| Parameter | Description |
| --- | --- |
| `{SECRET_ID}` | The `id` of the secret that you want to reauthorize. |

**Request**

The `data` object in the request payload must contain a `meta.action` property set to `reauthorize`.

```shell
curl -X PATCH \
  https://reactor.adobe.io/secrets/SEa3756b962e964fadb61e31df1f7dd5a3 \
  -H 'Authorization: Bearer {ACCESS_TOKEN}' \
  -H 'x-api-key: {API_KEY}' \
  -H 'x-gw-ims-org-id: {ORG_ID}' \
  -H 'Accept: application/vnd.api+json;revision=1' \
  -H 'Content-Type: application/vnd.api+json' \
  -d '{
        "data": {
          "attributes": {
            "type_of": "oauth2-google"
          },
          "meta": {
            "action": "reauthorize"
          },
          "id": "SEa3756b962e964fadb61e31df1f7dd5a3",
          "type": "secrets"
        }
      }'
```

**Response**

A successful response returns the details of the updated secret. From here, you must copy and paste the `meta.authorization_url` into a browser to complete the authorization process.

```json
  "data": {
```

## Delete a secret {#delete}

You can delete a secret by including its ID in the path of a DELETE request. This is a hard delete with an immediate effect and does not require a library republish.

This operation removes the secret from the environment it is related to and the underlying resource is deleted.

>[!WARNING]
>
>If you have any deployed rules that reference a deleted secret, those rules will immediately cease to function. Any data elements that reference this secret must be updated or removed afterward.

**API format**

```http
DELETE /secrets/{SECRET_ID}
```

| Parameter | Description |
| --- | --- |
| `{SECRET_ID}` | The ID of the secret that you want to delete. |

{style="table-layout:auto"}

**Request**

```shell
curl -X DELETE \
  https://reactor.adobe.io/secrets/SEa3756b962e964fadb61e31df1f7dd5a3 \
  -H 'Authorization: Bearer {ACCESS_TOKEN}' \
  -H 'x-api-key: {API_KEY}' \
  -H 'x-gw-ims-org-id: {ORG_ID}' \
  -H 'Accept: application/vnd.api+json;revision=1' \
  -H 'Content-Type: application/vnd.api+json'
```

**Response**

A successful response returns HTTP status 204 (No Content) and an empty response body, indicating that the secret has been deleted from the system.
